二、产品概述

2.1 一句话定义

一款常驻 Windows
桌面的虚拟宠物狗软件。用户领养一只狗狗、给它起名，通过喂食和玩耍与它互动；狗狗有等级、性别和情绪，会主动和主人搭话，偶尔还会『独自外出旅行』并寄回一张精美的明信片。

2.2 产品定位与核心体验

这个产品的灵魂不是『又一个电子宠物』，而是两个词：陪伴感 与 小惊喜。

- 陪伴感：狗狗一直在桌面角落，会主动问『主人你在干嘛』，让人觉得不是一个人在用电脑。

- 小惊喜：狗狗会自己『去新疆旅行』，几小时后寄回一张明信片------这是用户截图分享、产生传播的关键。

所有设计决策都应回到这两个词：能增强陪伴感或制造小惊喜的，优先做；不能的，往后放。

2.3 目标用户

中国大陆地区的 Windows
电脑用户，尤其是：喜欢萌宠、长时间使用电脑、对桌面有装饰需求、愿意在社交平台分享有趣内容的年轻人群。

2.4 设计基调（美术与交互）

你明确要求『精美、有趣、有艺术感』。文档建议确立以下基调，并贯穿全程：

- 画风统一：手绘 /
  扁平插画风格，温暖配色，避免廉价感。所有狗狗品种、明信片、UI
  必须出自同一画风。

- 动效细腻：狗狗待机时有呼吸起伏、偶尔眨眼、摇尾巴等小动作，细节决定『有没有生命感』。

- 交互克制：桌面宠物不能太吵。主动说话要有频率上限，不打扰用户工作。

清晰的边界是项目能完成的前提。第一版严格限定如下。

  -----------------------------------------------------------------------

  互动                喂食、玩耍两种核心互动

  本地存档            所有数据保存在本地，关机重开不丢失

+-----------------------------------------------------------------------+

|                                                                       |

|                                                                       |

3.3 为什么这样砍

『单机』的含义是：第一版不需要任何服务器。这一点至关重要------它意味着第一版没有服务器费用、不需要
ICP 备案、没有数据合规负担。

旅行明信片完全可以做成纯本地功能：狗狗触发外出后，软件在本地用打包好的风景素材生成一张明信片图片，过几小时本地弹出通知。用户体验和『真的从服务器寄来』几乎没有区别，但成本和合规负担天差地别。第一版务必走这条路。

四、狗狗核心设定

4.1 品种（第一版 6 个）

品种『足够多』是好目标，但每个品种都需要一整套美术资源。第一版控制在 6
个最受中国用户喜爱的常见品种，后续版本再扩充。

  -----------------------------------------------------------------------
  **品种**      **性格设定建议**             **气泡说话风格**
  ------------- ---------------------------- ----------------------------
  柴犬          高冷、有点傲娇               话少、偶尔毒舌

  柯基          活泼、爱卖萌                 话多、爱撒娇

  金毛          温柔、暖心                   贴心、爱关心主人

  哈士奇        二、精力旺盛                 跳脱、说怪话

  泰迪          黏人、敏感                   爱问主人在不在

  中华田园犬    忠诚、朴实                   实在、接地气
  -----------------------------------------------------------------------

说明：不同品种的『性格』主要体现在台词库和说话频率上，玩法机制可保持一致，以控制第一版工作量。

4.2 性别

领养时可选公 /
母。第一版性别主要影响：称呼（如『狗子』『狗妹』）、部分台词、可能的外观小差异（如配饰颜色）。不引入繁殖等复杂机制。

4.3 等级

狗狗通过互动累积『亲密度经验』升级。等级的作用是给用户长期目标感。

  -----------------------------------------------------------------------
  **项目**         **设定建议**
  ---------------- ------------------------------------------------------
  经验来源         喂食 +少量经验；玩耍 +较多经验；旅行归来 +一笔经验

  等级范围         第一版设 1-20 级，每级所需经验递增

  升级反馈         播放升级动画 + 气泡庆祝 +
                   可选解锁内容（如新台词、新表情）

  防沉迷考量       每日经验可设软上限，避免一次性刷满，鼓励每天来
  -----------------------------------------------------------------------

4.4 情绪

情绪是『陪伴感』的核心。狗狗的表情、动画、说话内容都由当前情绪决定。

  --------------------------------------------------------------------------
  **情绪状态**   **触发条件**                **表现**
  -------------- --------------------------- -------------------------------
  开心           刚被喂食/玩耍、亲密度高     活泼动画、摇尾巴、说开心的话

  普通           默认状态                    正常待机动画

  饿了           饥饿值低于阈值              无精打采动画、说想吃东西的话

  无聊           长时间没有互动              发呆/打哈欠动画、主动找主人玩

  想旅行         随机触发旅行前              看向远方、说想出去走走
  --------------------------------------------------------------------------

4.5 饥饿值

饥饿值随真实时间缓慢下降，提醒用户回来照顾狗狗。注意：第一版的饥饿设定应温和------狗狗不会因为没喂食而『生病』或产生负面惩罚，避免让用户有压力和愧疚感。这是装饰性陪伴软件，不是养成压力游戏。

五、旅行明信片系统（产品亮点）

这是整个产品最大的差异化亮点，也是社交传播的引爆点。第一版以『纯本地实现』为原则，单独成章详细说明。

5.1 完整流程

6.  触发：狗狗在『想旅行』情绪下，或每隔一段时间随机触发外出（建议频率不要太高，如
    2-4 天一次，保持稀缺感）。

7.  出发：狗狗气泡告别『我要去新疆玩几天啦，等我明信片！』，然后狗狗从桌面『离开』（淡出）。

8.  旅行中：狗狗暂时不在桌面。可在托盘图标或小窗口显示『旅行中』状态。旅行时长建议数小时（真实时间），不宜过长。

9.  寄回明信片：旅行结束，本地生成一张明信片图片，弹出系统通知『你的狗狗寄明信片回来啦』。

10. 查看：用户点击查看明信片------一张精美的卡片：风景背景 +
    狗狗在景点的小图 + 一段狗狗口吻的文字 + 邮戳/日期。

11. 归来：狗狗回到桌面（淡入），情绪为『开心』，并获得一笔经验。

12. 收藏与分享：明信片存入『明信片相册』，用户可随时翻看；可将明信片导出为图片保存到电脑，自行发到朋友圈。

5.2 明信片内容构成

  ----------------------------------------------------------------------------
  **元素**     **说明**                     **第一版实现方式**
  ------------ ---------------------------- ----------------------------------
  目的地       如新疆、西藏、云南、海边等   预设 8-12 个目的地，每个配 1-2
                                            张风景图

  风景背景     目的地的代表性风光           美术预先制作，打包在软件内

  狗狗形象     狗狗在景点的小图             可用狗狗通用形象叠加在风景上

  文字         狗狗口吻的一段话             每个目的地预设多条文案，随机抽取

  邮戳/日期    增加真实感                   代码动态生成当前日期
  ----------------------------------------------------------------------------

+----------------------------------------------------------------------------------------------------+
| **文案示例（供美术/文案参考，不要照抄）**                                                          |
|                                                                                                    |
| 『主人！新疆的天好蓝好蓝，蓝得像你上次没收的快递盒。我在这边吃到了好大的羊肉串，下次带你来。------ |
| 你的狗子』                                                                                         |
|                                                                                                    |
| 建议每个目的地准备 5-8 条这种温暖、略带幽默的文案，随机抽取，让用户多次旅行也不腻。                |
+----------------------------------------------------------------------------------------------------+

5.3 关于分享与微信（重要说明）

你的目标是『分享到朋友圈形成裂变』。需要明确技术现实：

- 桌面软件无法像手机 App
  那样『一键唤起微信分享』，那需要接入微信开放平台并完成企业认证，流程较长。

- 第一版的现实做法：把明信片做成一张足够精美、足够有传播欲的图片，提供『保存图片』按钮。用户保存后自己发朋友圈。

- 为提升传播：明信片图片角落可放一个小而精致的产品 logo
  或名字水印，让看到的人知道这是什么软件。

- 微信一键分享、邀请好友等裂变功能，待第二版接入服务器和微信开放平台后再做。

六、狗狗主动搭话系统

『狗狗会主动问你在干嘛』是陪伴感的关键。第一版用预设台词库实现，不接入真实
AI。

6.1 为什么第一版不用真 AI

  ----------------------------------------------------------------------------------------
  **方案**                   **优点**                 **缺点**
  -------------------------- ------------------------ ------------------------------------
  预设台词库（第一版选用）   免费、离线、稳定、可控   台词会重复

  接入大模型 API             对话生动、不重复         每次对话花钱、必须联网、有内容风险
  ----------------------------------------------------------------------------------------

结论：第一版用台词库把体验跑通。台词库写得足够丰富（按情绪、按品种、按时间段分类），重复感就能大幅降低。AI
对话留作未来增值功能。

6.2 台词库的组织方式

- 按情绪分类：开心 / 普通 / 饿了 / 无聊 各一组台词。

- 按品种性格分类：柴犬的话偏冷淡，柯基的话偏黏人（见 4.1）。

- 按时间段分类：早上『主人早呀』、深夜『这么晚还不睡？』、饭点『到饭点了，你吃了吗』。

- 按场景分类：旅行出发、旅行归来、升级时各有专属台词。

台词库建议做成独立的文本/JSON 文件，方便你随时增删，不用改代码。

6.3 说话频率（防打扰）

+-----------------------------------------------------------------------+
| **频率红线**                                                          |
|                                                                       |
| 桌面宠物最容易让人卸载的原因就是『太吵、打扰工作』。                  |
|                                                                       |
| 主动说话要有冷却时间（如至少间隔 10-20 分钟），并设每日上限。         |
|                                                                       |
| 提供一个『安静模式』开关，用户开会或专注时可临时让狗狗不说话。        |
+-----------------------------------------------------------------------+

八、性能与体验要求

Electron
内存偏高是现实，但可以通过以下方式尽量优化，开发每个模块时都应留意。

- 控制动画帧数：待机动画不需要高帧率，能流畅即可，帧数越多越占资源。

- 闲置降频：用户长时间没操作、狗狗在待机时，降低画面刷新频率。

- 关闭无用功能：Electron 里用不到的功能（如某些后台进程）按 Claude
  建议关闭。

- 图片资源压缩：所有美术 PNG 在打包前做压缩，减小体积和内存。

- 单实例运行：防止用户重复打开多个软件实例。

体验红线（务必遵守）：

- **不打扰：主动说话有冷却和上限，提供安静模式。**

- **不卡顿：拖动狗狗、播放动画要流畅，不能掉帧。**

- **不丢档：任何数据变化后及时存档，软件异常关闭也不丢狗狗。**

- **不施压：饥饿值等数值不做惩罚机制，保持轻松陪伴的基调。**

10.2 软件著作权

在中国大陆，软件上架应用商店通常需要『软件著作权登记证书』。这个证书的申请需要一定时间，建议在开发中后期就着手准备，不要等到全部做完才办，以免卡住发布节奏。

10.3 关于联网与备案（第二版才涉及）

第一版是纯单机，不涉及备案。但当你开始做第二版（用户互动、微信裂变、服务器明信片）时，会涉及：

- 服务器：需要持续的服务器、域名费用。

- ICP 备案：在中国大陆提供联网服务通常需要 ICP
  备案，需提前办理、需要时间。

- 微信开放平台：一键分享/裂变需接入微信开放平台，通常需要企业资质和认证。

- 个人信息合规：一旦收集用户数据，需遵守个人信息保护相关法规，做好隐私政策。

这些都不影响第一版，但建议你心里提前有数，便于规划第二版的预算和时间。

10.4 内容安全

狗狗的台词、明信片文案等所有面向用户的文字内容，应保持友好、积极、健康，避免任何不当内容。这既是产品调性的要求，也是上架审核的要求。
